**Unsupervised Anomaly Detection in SDO/AIA Multi-channel Imagery**

## Overview

The **SOLAR** project addresses the critical challenge of analyzing the immense and complex dataset generated by the Atmospheric Imaging Assembly (`AIA`) aboard NASA's Solar Dynamics Observatory (`SDO`). Traditional methods often focus on well-characterized, energetic events (flares, CMEs) or rely on manual inspection, potentially overlooking subtle, rare, or entirely novel phenomena hidden within the petabytes of observational data.

This project pioneers an **unsupervised machine learning approach** to automatically detect anomalous patterns in multi-channel EUV/UV images from SDO/AIA. By learning the 'normal' behavior directly from the data distribution across multiple wavelengths simultaneously, SOLAR aims to:

1.  **Identify statistically significant deviations** without pre-defined templates or labels.
2.  **Flag potentially interesting regions or time periods** that warrant deeper investigation by domain experts.
3.  **Move beyond catalog-driven analysis** towards data-driven discovery of subtle precursors, faint structures, or uncharacterized solar processes.
4.  Provide a **modular and extensible computational framework** for anomaly exploration in solar physics.

This work represents a novel application of unsupervised techniques to the specific challenge of anomaly discovery within the high-dimensional feature space defined by SDO/AIA's multi-channel observations.

## Features

*   **Data Handling & Versioning:** Ingests standard SDO/AIA Level 1.5 FITS files. Manages core datasets using [DVC](https://dvc.org/) with Google Cloud Storage (GCS) backend for reproducibility and collaboration.
*   **Preprocessing Pipeline:**
    *   Automatic solar disk masking using FITS metadata.
    *   Configurable image downsampling (e.g., to 512x512) for computational tractability.
    *   Robust inter-channel intensity normalization using `RobustScaler` to handle outliers (e.g., flares).
    *   Handling of off-limb/missing pixels (`NaN`).

    **Additional documentation**: See the [API Reference](API\README.md).

*   **Unsupervised Anomaly Detection:** Utilizes `Isolation Forest` to assign anomaly scores based on multi-channel pixel intensities.
*   **Anomaly Characterization:** Applies clustering algorithms (`K-Means`, `MiniBatch K-Means`, `GMM`) to group detected anomalies based on their spectral signatures.
*   **Visualization:** Generates multi-panel plots overlaying detected anomaly clusters onto the original AIA channel images for intuitive interpretation.
*   **Modular Pipeline:** Structured Python code allowing for experimentation and extension.

## Methodology Summary

The core analysis pipeline implemented follows these logical steps:

1.  **Data Loading:** Loads specified AIA channel FITS files (expected to be present locally, e.g., via `dvc pull`). Applies disk mask and resizes.
2.  **Preprocessing:** Stacks channels, reshapes data into a (pixels x channels) matrix, removes invalid pixels, and applies `RobustScaler` normalization.
3.  **Anomaly Scoring:** Fits an `Isolation Forest` model to the normalized pixel data and calculates anomaly scores for each valid pixel.
4.  **Anomaly Thresholding:** Identifies anomalous pixels based on a user-defined `anomaly_threshold` applied to the scores.
5.  **Anomaly Clustering:** Applies a clustering algorithm (e.g., `MiniBatchKMeans`) to the feature vectors of the anomalous pixels to group them into `k` clusters.
6.  **Visualization:** Maps the cluster labels back to the 2D image grid and generates plots showing clusters overlaid on the original channel images.

## Installation and Data Access

Follow these steps to set up the project environment and access the versioned data using DVC and Google Cloud Platform (GCP).

**Prerequisites:**
*   Git
*   Python (3.8+ recommended)
*   Access to the project's GCP Service Account JSON key file.

**Steps:**

1.  **Clone the Repository:**
    ```bash
    git clone https://github.com/AlyonaCIA/SOLAR.git # Adjust URL if needed
    cd SOLAR
    ```

2.  **Create and Activate Virtual Environment:**
    *   **macOS/Linux:**
        ```bash
        python3 -m venv .venv
        source .venv/bin/activate
        ```
    *   **Windows (PowerShell/Git Bash):**
        ```bash
        python -m venv .venv
        .venv\Scripts\activate
        ```

3.  **Install Core Dependencies:**
    ```bash
    pip install -r requirements.txt
    ```
    *(This should include libraries like `numpy`, `scikit-learn`, `astropy`, `sunpy`, `matplotlib`, `scikit-image`, etc.)*

4.  **Install DVC with Google Cloud Support:**
    ```bash
    pip install "dvc[gcs]"
    ```

5.  **Configure GCP Access (Service Account - Recommended):**
    Access to the DVC-managed data requires authentication via a specific GCP service account.
    *   Obtain the service account `.json` key file provided by the project maintainers.
    *   **IMPORTANT SECURITY NOTE:** **NEVER** commit this `.json` key file to Git. Ensure it is listed in your `.gitignore` file.
    *   Set the environment variable `GOOGLE_APPLICATION_CREDENTIALS` to point to the path of your key file.
        *   **macOS/Linux:**
            ```bash
            export GOOGLE_APPLICATION_CREDENTIALS="/path/to/your/gcp-key.json"
            ```
            *(Add this line to your shell profile (e.g., `.zshrc`, `.bashrc`, `.bash_profile`) for persistence across sessions).*
        *   **Windows (PowerShell):**
            ```powershell
            $env:GOOGLE_APPLICATION_CREDENTIALS="/path/to/your/gcp-key.json"
            ```
            *(You might need to set this as a persistent environment variable via system settings).*

    *   The service account associated with the key file needs at least the `roles/storage.objectViewer` permission on the project's GCP bucket (`gs://ml-project-dvc-bucket`).
    *   *Note:* Direct `gcloud auth login` is generally not required when using a service account key file via the environment variable for DVC operations.

6.  **Download DVC-Managed Data:**
    Once the environment is activated and the GCP credentials environment variable is set, pull the versioned data:
    ```bash
    dvc pull
    ```
    This command downloads the data files tracked by DVC from the GCP remote storage (`gcp-remote` defined in `.dvc/config`) to their corresponding locations within the project structure (e.g., `./sdo_data/`).

7.  **Verify Data:**
    Check if the expected data directories/files are now present:
    ```bash
    ls ./sdo_data/
    # Should show subdirectories like aia_94, aia_171, etc.
    ```

8.  **macOS Specific - SSL Certificates (If needed):**
    On some macOS setups, Python might require updated SSL certificates for certain network operations (like downloading FITS files if using the alternative script below). If you encounter SSL errors:
    ```bash
    /Applications/Python\ 3.x/Install\ Certificates.command
    # Replace 3.x with your Python version (e.g., 3.9, 3.10)
    ```

9.  **(Optional) Google Cloud SDK (`gcloud`) Installation:**
    While not strictly necessary for *pulling* data using a service account key, installing the `gcloud` CLI is useful for managing GCP resources or if direct interaction with the bucket is needed.
    *   **macOS (x86_64):** Download and run the installer as described in the official [Google Cloud SDK documentation](https://cloud.google.com/sdk/docs/install). The `.tar.gz` method involves `tar -xvzf ...`, `cd google-cloud-sdk`, `./install.sh`, and potentially updating your shell profile (`.zshrc`, `.bashrc`).
    *   **Ubuntu/Debian:** `sudo apt-get update && sudo apt-get install google-cloud-sdk`
    *   Follow `gcloud init` if performing initial setup or managing configurations.

Discussion and Limitations
Unsupervised Evaluation: Quantitative validation remains challenging without ground truth. Current evaluation relies on visual inspection, coherence, and physical plausibility.
Physical Interpretation: Linking statistical anomalies/clusters to physical processes requires domain expertise and further analysis.
Hyperparameter Sensitivity: Performance depends significantly on anomaly_threshold, n_clusters, etc. Systematic tuning is needed.
Computational Cost: High-resolution/long-duration analysis is demanding.
Artifact Handling: Residual instrumental effects can be misidentified.
